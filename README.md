## 📌 Как пользоваться роадмапом

Каждый этап построен по схеме:

```
что учим → зачем это нужно → где брать материал → какой проект → как проверить себя
```

**3 железных правила:**

1. **Не прыгай вперёд.** Go обманчиво простой — без фундамента упрёшься в потолок на собесах.
2. **Пиши код сам.** Подсмотреть можно только после часа борьбы с задачей.
3. **Делай проекты до конца.** Незаконченный проект учит хуже законченного «плохого».

## 🟢 Этап 0. Фундамент (2–3 недели)

Если ты вообще не программировал — этот этап обязательный. Если знаешь любой язык, проскакивай к Этапу 1.

**Темы:**
- как работает компьютер на пальцах: память, стек, куча, процессы и потоки;
- что такое компилируемый язык и чем он отличается от интерпретируемого;
- системы счисления, базовая алгебра логики;
- терминал, базовые команды bash/zsh;
- Git и GitHub: clone, commit, push, pull, branch, merge, rebase, PR.

**Материалы:**
- 📘 «Грокаем алгоритмы» — Адитья Бхаргава;
- 🎓 курс «Введение в программирование» от Хекслет (бесплатно);
- 📕 Pro Git (бесплатная книга на русском, git-scm.com/book/ru);
- 🎥 MIT «Missing Semester» — терминал, Git, инструменты разработчика.

**✅ Чек-пойнт:** можешь склонировать репо, создать ветку, сделать коммит, открыть PR и решить merge-конфликт без паники.

## 🟢 Этап 1. Синтаксис и основы Go (3–4 недели)

Здесь учим читать и писать идиоматичный Go-код. Не пиши на Go как на Python или Java — он другой по философии: меньше магии, больше явности.

**Темы:**
- установка Go, GOPATH vs Go modules, структура проекта;
- переменные, типы, константы, iota;
- управляющие конструкции: if, for, switch (особенно type switch);
- функции, множественный возврат, named returns, variadic;
- указатели — без них дальше никуда;
- структуры, методы, получатели по значению vs по указателю;
- интерфейсы — главная фишка Go, учить долго и вдумчиво;
- error handling: `errors.Is`, `errors.As`, обёртывание через `%w`;
- срезы, массивы, мапы (как устроены внутри);
- panic, recover, defer и порядок их выполнения.

**Материалы:**
- 🎓 A Tour of Go (go.dev/tour) — обязательно пройти полностью;
- 📘 «Go на практике» — Алан Донован, Брайан Керниган;
- 🎓 курс «Программирование на Go» от Stepik (бесплатно, на русском);
- 🎥 каналы Никиты Соболева и Алекса Карпова на YouTube;
- 🔗 gobyexample.com — короткие примеры на каждый чих.

**🛠️ Проект:** CLI-утилита для управления todo-списком с сохранением в JSON, флагами через `flag`, цветным выводом и тестами.

**✅ Чек-пойнт:** можешь объяснить разницу между срезом и массивом, что произойдёт при `append` к срезу с `cap == len`, и когда метод нужно делать на указателе.

## 🟡 Этап 2. Глубокий Go (4–6 недель)

Этот этап отделяет Junior от Middle. Здесь учим, как Go устроен изнутри.

**Темы:**
- memory model: stack vs heap, escape analysis (`go build -gcflags="-m"`);
- как устроены slice, map, string, interface на уровне runtime;
- сборщик мусора (GC): tricolor marking, write barriers, GOGC, GOMEMLIMIT;
- горутины и планировщик (GMP-модель);
- generics (type parameters, constraints) — стандарт с 1.18;
- reflection и unsafe — знать, чтобы не использовать без причины;
- embedding структур и интерфейсов вместо наследования;
- идиомы: functional options, builder, errors as values.

**Материалы:**
- 📘 «100 Go Mistakes and How to Avoid Them» — Тейва Харсаньи (must-read);
- 📝 блог Дэйва Чейни (dave.cheney.net);
- 🎥 доклады GopherCon, FOSDEM Go Devroom (YouTube);
- 📝 статьи Уильяма Кеннеди (Ardan Labs).

**🛠️ Проект:** in-memory key-value store с TTL, конкурентный, с benchmark-тестами и профилированием через pprof.

**✅ Чек-пойнт:** объясняешь escape analysis, почему интерфейс — это два указателя, и когда стоит/не стоит использовать reflection.

## 🟡 Этап 3. Конкурентность (3–4 недели)

Главная причина, почему Go выбирают для бэкенда. Здесь живут самые жёсткие баги.

**Темы:**
- горутины: запуск, жизненный цикл, утечки;
- каналы: буферизованные и небуферизованные, направленные;
- select, default, timeout-паттерн;
- sync: Mutex, RWMutex, WaitGroup, Once, Cond, Map, Pool;
- context: cancel, deadline, value — как правильно прокидывать;
- паттерны: pipeline, fan-in/fan-out, worker pool, semaphore;
- race detector (`go test -race`), data race vs race condition;
- atomic-операции, sync/atomic, atomic.Pointer.

**Материалы:**
- 📘 «Concurrency in Go» — Кэтрин Кокс-Будэй (лучшая книга по теме);
- 🎥 «Go Concurrency Patterns» — Роб Пайк;
- 🎓 курс «Concurrency in Go» на Stepik;
- 📝 блог-серия Влада Михальчука по конкурентности (на русском).

**🛠️ Проект:** многопоточный веб-краулер с rate limiting, retry, отменой через context и graceful shutdown.

**✅ Чек-пойнт:** без подсказок реализуешь worker pool, объясняешь deadlock vs livelock, понимаешь когда канал лучше мьютекса.

## ⚡ Шпаргалка по командам Go

```bash
# Модули
go mod init github.com/user/project    # создать модуль
go mod tidy                             # подтянуть зависимости
go mod vendor                           # положить зависимости в ./vendor
go mod why <pkg>                        # почему подтягивается зависимость

# Сборка и запуск
go run .                                # запустить
go build -o bin/app                     # собрать
go install                              # собрать и положить в $GOBIN

# Тесты и бенчмарки
go test ./...                           # все тесты в проекте
go test -race ./...                     # с race detector
go test -run TestName -v                # один тест
go test -bench=. -benchmem              # бенчмарки
go test -cover                          # покрытие
go test -fuzz=FuzzName                  # фаззинг

# Профилирование
go tool pprof cpu.prof
go test -cpuprofile=cpu.prof -memprofile=mem.prof

# Качество кода
go fmt ./...                            # форматирование
go vet ./...                            # статический анализ
golangci-lint run                       # все линтеры разом

# Полезное
go doc fmt.Println                      # документация по функции
go env                                  # переменные окружения
go version                              # версия
```

## 🪤 Типичные ошибки новичков

| Ошибка | Что не так | Как правильно |
|--------|-----------|---------------|
| Захват переменной цикла в горутине | До Go 1.22 переменная одна на все итерации | Передавай как аргумент или используй Go 1.22+ |
| Сравнение интерфейса с nil | Интерфейс не равен nil, если у него есть тип, но значение nil | Сравнивай конкретный тип, не интерфейс |
| Игнорирование ошибок | `_ = doSomething()` — выстрел в ногу | Обрабатывай или хотя бы логируй |
| Утечка горутин | Запустил и не отменил | Всегда передавай context с cancel |
| Мутация мапы из горутин | Data race и паника | sync.Mutex или sync.Map |
| Закрытие канала со стороны получателя | Паника | Закрывает только отправитель |
| `defer` в цикле | Откладывает до конца функции, а не итерации | Вынеси в отдельную функцию |
| append без записи результата | `append(s, x)` без `s = append(...)` | Всегда присваивай результат |
| Большая структура по значению | Лишние копии в памяти | Передавай по указателю |
| Использование reflection без причины | Медленно и хрупко | 99% случаев решается без него |

## ❓ FAQ

**С какого этапа можно идти на Junior-собес?**
После 1–3 этапов + один доделанный pet-проект (CRUD API с БД). Если есть опыт в другом языке — даже раньше.

**Нужно ли учить алгоритмы для Go-вакансий?**
В большинстве продуктовых компаний — да, на уровне easy/medium LeetCode. В стартапах и геймдеве часто не спрашивают совсем.

**Go или Rust в 2026?**
Разные ниши. Go — бэкенд, инфраструктура, CLI, быстрая разработка. Rust — там, где нужен максимум производительности и контроль над памятью. Учить Rust имеет смысл *после* Go, а не вместо.

**Какой фреймворк выбрать для веба?**
Для обучения — стандартный `net/http` + chi. Для прод — то, что используется в компании. `gin`, `echo`, `fiber` — все рабочие.

**Стоит ли учить ORM?**
GORM знать полезно для совместимости, но в продакшне всё больше выбирают **sqlc** или чистый pgx. SQL знать обязательно.

**Сколько проектов нужно в портфолио?**
3–4 законченных проекта разной сложности. Лучше один большой и доделанный, чем десять брошенных.

**Когда менять работу для роста?**
Если 6+ месяцев нет новых задач и не растут компетенции — пора смотреть варианты.
